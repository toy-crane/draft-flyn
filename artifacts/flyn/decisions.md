# Flyn — 디자인 결정 기록

> AI와 카톡처럼 영어로 채팅하면서, 내가 보낸 메시지가 맞는지 틀린지 바로 확인할 수 있는 앱
>
> 작성일: 2026-05-21
> 출처: `/grill-me` 인터뷰 결과

---

## 0. 제품 요약

### 문제
ChatGPT로 영어 대화 연습을 해보면, 사용자가 "I go to school yesterday"라고 써도 ChatGPT는 아무 말 없이 대화를 이어간다. 내가 틀렸는지조차 모른다. 그렇다고 매번 "내 문장 틀렸어?" 물으면 대화 흐름이 끊긴다.

**대화를 이어가면 뭘 틀렸는지 모르고, 멈춰서 확인하면 흐름이 끊긴다.**

### 해결
카카오톡처럼 익숙한 채팅 UI 위에 4가지 기능을 얹는다:

1. **실시간 교정** — 메시지 옆에 ✓/✱/💡 마크가 즉시 표시. 흐름 끊김 없음.
2. **시나리오 기반 대화** — 폼으로 상황·역할 정의 → AI가 페르소나로 대응
3. **더 알아보기** — 교정에 대해 별도 창에서 질문 가능
4. **한국어 → 영어 변환** — 영어가 안 떠오르면 한국어로 써도 됨

| 항목 | 결정 |

| **필드별 ✨ 자동 채움** | 각 필드 옆 ✨ 버튼. 다른 필드가 채워져 있으면 컨텍스트 반영. |
| **전체 ✨** | 폼 헤더에 별도 버튼 ("AI에게 전부 맡기기") |
| **필드 길이** | 각 1줄 (~40자), 메모만 2-3줄 |

| **취소 시** | "이 시나리오를 저장하지 않을까요?" 확인 모달 | — |

## 5. 시나리오 (목표)

| 항목 | 결정 | 근거 |
|---|---|---|
| **목표 생성** | 폼 확정 후 AI가 3개 자동 생성 | 사용자 입력 부담 ↓ |
| **편집 가능성** | 재생성(↻)만 가능. 직접 편집 X | 사용자가 모호하게 수정하면 달성 평가 불가. 모델 복잡도 ↑ |
| **노출 시점** | 폼 제출 → 로딩 → 미리보기 → "시작하기" | 사용자 검증 기회 제공 |
| **카드 구조** | 짧은 영어 행동 + 한국어 부연 (예: "Order a latte" / "음료 주문하기") | — |
| **시나리오 요약** | 목표와 함께 1줄 요약 표시 (예: "Bluebird Coffee에서 손님으로 라떼 주문하기") | 사용자가 자기 입력의 해석 확인 |
| **생성 실패 시** | "다시 시도" 또는 "목표 없이 시작" 옵션 | 목표 없이 시작 시 종료는 수동 |

### 목표 표시 (채팅 중)

| 항목 | 결정 | 근거 |
|---|---|---|
| **표시 위치** | 채팅 헤더 아래 항상 표시 | 동기부여 + 방향 제시. 숨기면 길을 잃음. |
| **시각화** | 3개 체크박스 형태 (⚪/✓), 짧은 텍스트 (예: "1. 음료 주문하기 ⚪") | — |
| **평가 시점** | 매 사용자 메시지마다 실시간 (분석 호출에 포함) | 즉각 피드백 = 학습 동기 |
| **달성 시 UI** | 200ms 펄스/색 변화 (짧고 거슬리지 않게) | — |
| **3개 모두 달성 시** | "목표 다 했어요! 끝낼까요?" 권유 모달. "끝내기" / "더 대화" 선택 | 자동 종료는 부당한 느낌. 사용자 통제권 유지 |

## 8. 교정 시스템

### 마크 정의

| 마크 | 의미 | 발동 조건 |
|---|---|---|
| ✓ OK | 문제없음 | 원어민이 자연스럽게 받아들이는 문장 |
| ✱ 교정 필요 | 틀렸거나 어색 | 원어민이 "어?" 하고 멈칫할 만한 오류 |
| 💡 대안 제시 | 더 자연스러운 표현 | 의미 전달은 OK, textbook스럽거나 약간 어색 |
| 🌐 번역 | 한→영 변환 결과 | 한국어 입력 시 |

### 임계점

| 항목 | 결정 | 근거 |
|---|---|---|
| **기준** | 이해 가능성 (원어민 기준) | 학원 선생님 수준 엄격함은 의욕 ↓ |
| **메시지당 마크** | 하나만 | 한 버블에 ✱ + 💡 동시 X. 오류가 무거우면 ✱가 이김. |
| **여러 오류** | ✱ 하나, 펼친 패널에 리스트 | — |
| **마이너 오타** | 무시 (✓로 통과) | 학습 가치 없음 |

### 교정 타이밍

| 항목 | 결정 |
|---|---|
| **동시성** | 채팅 응답과 **병렬 스트리밍** |
| **표시** | ✱는 **접힌 상태로** 사용자 메시지 옆에 붙음 |
| **펼침** | 사용자가 탭하면 같은 버블 내 섹션이 펼쳐짐 |

## 9. 버블 레이아웃 (Option B — 라벨드 섹션)

### 핵심 원칙
- **사용자 입력 = 메인** (변형 없이 그대로 보존)
- **AI 출력(번역/교정) = 부수** (라벨드 섹션, 기본 펼침)
- **한→영과 교정은 동일 포맷, 아이콘으로만 구분**

### 케이스별 레이아웃

```
[Case 1: 영어 입력 + 교정]
┌────────────────────────────────┐
│ I go to school yesterday       │ ← 사용자 입력
│                                │
│ ✱ 교정                    [⌃] │ ← 섹션 헤더 (기본 펼침)
│ I went to school yesterday     │
│ "went"이 더 자연스러워요       │
└────────────────────────────────┘

[Case 2: 한글 입력 → 영어 변환]
┌────────────────────────────────┐
│ 학교 갔어                       │ ← 사용자 입력 (한글 원본)
│                                │
│ 🌐 번역                   [⌃] │
│ I went to school yesterday     │
└────────────────────────────────┘

[Case 3: 접은 상태]
┌────────────────────────────────┐
│ I go to school yesterday       │
│ ✱ 교정                    [⌄] │
└────────────────────────────────┘

[Case 4: ✓ OK]
┌────────────────────────────────┐
│ I had a great time today       │ ← 부수 영역 없음
└────────────────────────────────┘
```

### 세부

| 항목 | 결정 |
|---|---|
| **접기 토글** | 우측 상단 [⌃/⌄]. 메시지별로 상태 유지. |
| **여러 섹션 동시** | 위에서 아래 의존 순서 (예: 🌐 다음 💡) |
| **시각 위계** | 사용자 입력은 정상 톤, 섹션 헤더는 작고 회색 |

## 10. 한국어 → 영어 변환

| 항목 | 결정 | 근거 |
|---|---|---|
| **트리거** | 자동 감지 (한글 타이핑 시) | 토글은 마찰 ↑. 안전망 접근에 또 버튼 누르는 건 모순. |
| **표시 시점** | 전송 **전** 미리보기 | 사용자가 영어 번역을 학습 자료로 의식적으로 봄. 전송 후 치환은 휙 지나침. |
| **미리보기 UI** | 입력창 위 카드 ("→ I went to school"). ↗ 또는 "이걸로 보내기" 탭하여 전송 | — |
| **버블 표시** | 한글 원본 + 영어 번역 둘 다 보존 (Q9의 라벨드 섹션 사용) | — |
| **AI가 보는 것** | 영어 메시지만 (한글 안 봄) | Q7의 "AI는 원본을 본다"와 일관성. 단, 한국어 입력은 영어로 변환된 게 "원본"으로 간주. |
| **교정 마크** | 번역된 메시지에는 ✱/💡 없음 (🌐만) | 본인이 쓴 게 아니므로 교정 대상 X |

## 11. 더 알아보기 (사이드 대화)

| 항목 | 결정 | 근거 |
|---|---|---|
| **트리거** | 펼쳐진 교정 섹션 안에 "더 알아보기 →" 버튼 (또는 교정 카드 길게 누르기) | — |
| **UI (웹)** | 오른쪽 슬라이드인 드로어 (overlay, 400-460px) | GitHub Copilot Chat, Linear AI, Notion AI 컨벤션. 사용자 학습 비용 0. |
| **UI (모바일)** | 바텀시트 (50-70% 높이) | 반응형 |
| **헤더** | 작은 카드로 "맥락 중인 교정" 표시 | 사용자가 어떤 교정에 대해 묻는지 항상 보임 |
| **닫기** | X 버튼 + Esc 키. **click-outside는 비활성화** | 실수 방지 |
| **시드 메시지** | 시트 열릴 때 "이 교정에 대해 궁금한 점이 있나요?" + 자주 묻는 질문 칩 3개 ("왜 이게 틀려?", "다른 표현은?", "비슷한 예시") | — |
| **컨텍스트** | **메인 대화 전체** (시나리오 폼 + 모든 턴 + 해당 교정) | 적절한 답변 위해 대화 흐름 필요. 시나리오는 짧아 비용 미미. |
| **AI 페르소나** | 별도 — **한국어로 답하는 언어 코치** | 메인 대화의 캐릭터(바리스타 등)와 분리. 영어 예시 인용은 영어 그대로. |
| **메인 대화 영향** | 사이드 대화는 메인에 영향 X (완전 격리) | — |
| **영속성** | 일회용 (X/Esc로 명시 종료, 그 후 소실) | 같은 ✱ 다시 열어 같은 질문 또 하는 행동은 드묾 |

shadcn 컴포넌트: `<Sheet side="right">` 직접 활용 가능.

## 13. LLM 호출 아키텍처

### 매 사용자 메시지마다 필요한 작업

1. **채팅 응답** (시나리오 페르소나로 답변) — 스트리밍
2. **교정 분석** (✓/✱/💡 + 설명)
3. **목표 평가** (3개 중 달성된 게 있는가)

### 구조

| 호출 | 모델 힌트 | 출력 형식 | 비고 |
|---|---|---|---|
| **호출 1 (채팅)** | 빠른 모델 (GPT-4o, Claude Haiku 4.5, Gemini Flash) | 스트리밍 텍스트 | 사용자 원본 메시지 + 시나리오 컨텍스트 |
| **호출 2 (분석)** | 정확한 모델 (Claude Sonnet 4.6, GPT-4) | Structured JSON | 교정 + 번역 + 목표 평가 통합 |

두 호출은 **병렬**로 시작.

### 분석 호출 출력 예

```json
{
  "correction": {
    "status": "needs_correction",
    "icon": "✱",
    "corrected_text": "I went to school yesterday",
    "explanation": "'went'이 더 자연스러워요..."
  },
  "translation": null,
  "goals_achieved": [1]
}
```

`status`: `ok` | `needs_correction` | `alternative`
`translation`: 한국어 입력일 때만 채워짐 (실제로는 전송 전 변환이라 분석 호출엔 거의 안 들어옴)
`goals_achieved`: 이번 메시지로 달성된 목표 인덱스 (없으면 `[]`)

### 사이드 대화 ("더 알아보기")
- 별도 호출 (드로어 열릴 때만)
- 메인 대화 전체 컨텍스트 + 해당 교정 + 사용자 질문
- 시스템 프롬프트: "한국어로 답하는 언어 코치"

### 한→영 변환
- 사용자가 한글 타이핑 → 디바운싱 후 별도 호출
- 미리보기 카드 표시
- 전송 시점에는 영어만 보냄 (변환 호출은 이미 끝남)

### Latency 모델 (호출 2개 방식)

| 시점 | 이벤트 |
|---|---|
| 0ms | 사용자 전송 |
| 200-500ms | 채팅 응답 첫 토큰 (스트리밍 시작) |
| 1.5-3s | 분석 호출 완료 → ✱ 마크 + 목표 체크 표시 |

### 실패 처리

| 실패 | 처리 |
|---|---|
| 채팅 호출 실패 | "다시 시도" 버튼. 사용자 메시지 보존. |
| 분석 호출 실패 | 채팅은 정상 진행. 마크 없이(✓처럼) 조용한 fallback. |
| 둘 다 실패 | "다시 시도" + 사용자 메시지 보존. |
| 타임아웃 | 채팅 30초, 분석 10초 |

## 14. 데이터 모델 (개략)

```
User
 ├── id (Google OAuth)
 └── created_at

Scenario
 ├── id
 ├── user_id (FK)
 ├── situation       (텍스트)
 ├── their_role      (텍스트)
 ├── my_role         (텍스트)
 ├── memo            (텍스트, nullable)
 ├── goals           (JSON 배열 of {id, en, ko})
 ├── created_at
 └── updated_at

Conversation
 ├── id
 ├── scenario_id (FK)
 ├── status        ("in_progress" | "completed")
 ├── goals_achieved (JSON 배열 of goal index)
 ├── created_at
 ├── completed_at  (nullable)
 └── messages: Message[]

Message
 ├── id
 ├── conversation_id (FK)
 ├── role            ("user" | "assistant")
 ├── original_text   (사용자가 친 그대로 — 영어 또는 한글)
 ├── english_text    (한글이었으면 변환된 영어, 영어였으면 동일)
 ├── correction      (JSON: {status, icon, corrected_text, explanation} | null)
 ├── goals_achieved  (이 메시지로 달성된 목표, JSON 배열)
 └── created_at
```

> `original_text`와 `english_text`를 분리 보관하는 이유: 한글 입력 시 두 값이 다름. 버블 렌더링 시 둘 다 필요.
